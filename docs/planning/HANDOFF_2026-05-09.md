# ThreadView — handoff 2026-05-09

## What this project is

A Chrome MV3 extension that gives the user a clean reading view of any Gmail
thread. The architecture taps Gmail's own server-rendered print template
(`?view=pt&permthid=...`) instead of (a) scraping the live Gmail SPA's DOM
or (b) using the Gmail REST API. This sidesteps OAuth, Google verification,
CASA, and the SPA's rotating obfuscated class names.

## Status: v0.2.0 reading UI works end-to-end

End-to-end pipeline verified on three different Gmail threads in one Workspace
account today:

1. Content script on `mail.google.com` mounts a floating "ThreadView" button
   in a Shadow DOM root.
2. On click it reads `accountIndex` from the URL pathname, `permthid` from
   `<h2 data-thread-perm-id>` in Gmail's DOM, and `ik` from the resource
   timing API. Sends them to the background worker.
3. Background worker opens `chrome-extension://...viewer.html?...` in a new
   tab.
4. Viewer asks the background worker (NOT itself — see "Sharp edges" below)
   to fetch the Gmail print-view URL with credentials. Background returns
   the HTML.
5. Viewer parses with `parsePrintView()` — uses `<table class="message">`
   as the per-message anchor and `<div style="overflow: hidden;">` for body.
6. Viewer renders each message as a card via `renderMessage()` →
   `renderBody()`. Sanitizes HTML (allowlist-style), collapses quoted
   history into "Show quoted history ▾" toggles, opens external links in
   new tab.

Subject + date + sender + recipients + body all render correctly. Quoted
history collapsing identifies `<blockquote class="gmail_quote">`,
`div[id*="mail-editor-reference-message-container"]`,
`div[id*="divRplyFwdMsg"]`, and Gmail's print-marker
`<font>[Quoted text hidden]</font>`.

## Sharp edges that bit us today (READ THIS BEFORE EDITING)

These were all real bugs that took an hour each to diagnose. Don't re-discover them.

1. **Gmail's own service worker on mail.google.com intercepts every fetch
   to that origin.** Direct `fetch()` from the viewer page (which is
   `chrome-extension://`) gets eaten by Gmail's `sw.js?offline_allowed=1`
   and rejected with a network error from `fetchhandler.js`. Solution:
   ALL fetches to mail.google.com must go through the BACKGROUND service
   worker, which Gmail's SW cannot intercept. This is why we have
   `THREADVIEW_FETCH_PRINT` and `THREADVIEW_FETCH_IMAGE` in `sw.js`.
   Do not "simplify" by fetching directly from the viewer.

2. **`ik` is not always in the DOM.** Earlier today the `<img src="...?ik=">`
   approach worked. Later in the same session, on a different Gmail UI
   state, no element in the entire DOM had `ik=` in any attribute. The
   reliable source is `performance.getEntriesByType('resource')` —
   every recent Gmail XHR carries `ik=` in its URL, and resource timing
   remembers them. This is the FIRST approach in `readIkFromDom` for that
   reason; the DOM scans are fallbacks only.

3. **`web_accessible_resources` MUST include `src/content/inject.css`**, not
   just the viewer files. The content script loads its CSS by URL via
   `chrome.runtime.getURL("src/content/inject.css")` into a Shadow DOM,
   and Chrome treats that as "loaded by a page outside the extension"
   (because Shadow DOM is attached to Gmail's body). Without WAR,
   the floating button is invisible because its CSS is denied.

4. **Two-step reload after extension changes.** When you change ANY content
   script file (`inject.js`) or any file the content script loads
   (`inject.css`), you must (a) reload the extension at
   `chrome://extensions` AND (b) reload the Gmail tab. Without (b), Gmail
   has the OLD content script running, which throws "Extension context
   invalidated" on every button click. Background-only changes need only
   (a). Viewer-only changes need only (a) + open a fresh viewer tab.

5. **Each reply contains the full thread quoted underneath.** The parser
   gives faithful `bodyHtml` per message, which means the same content
   appears N times across N messages. The reading UI handles this by
   collapsing quoted history. Don't be surprised if a "small" 6-message
   thread is 263 KB of HTML.

6. **Print-view URL parameters that worked today (verified 2026-05-09):**
   `https://mail.google.com/mail/u/<N>/?ik=<key>&view=pt&search=all&permthid=thread-f:<digits>`
   Note `permthid` is `thread-f:<digits>` literally — the colon does NOT
   need to be URL-encoded (we let URLSearchParams encode it as `%3A` and
   Gmail accepts both forms). `simpl=msg-f:<id>` per-message params are
   optional; Gmail returns the full thread without them.

## Decisions worth flagging

- **No DOMPurify dependency.** Hand-rolled allowlist sanitizer in
  `renderBody.js`. If we ever ship to a real audience, swap for DOMPurify
  — it's hardened against edge cases (mutated XSS, mXSS, etc.) we
  haven't audited.
- **Images fetched as data URLs, not blob URLs.** Slightly wasteful for
  the message-passing channel (~33% base64 expansion) but simpler than
  managing blob URL lifetimes across the bg/viewer boundary. Cap is 10 MB
  per image. If users have huge inline images, raise cap or switch to
  blob URLs created via `chrome.runtime.getURL` + a fetch handler.
- **Floating button placement (bottom-right) over toolbar injection.**
  The plan called this out. Toolbar injection is a v3+ concern.
- **Single-account support.** `accountIndex` is read per-click from the
  Gmail tab the user is on. Multi-account works because each Gmail tab
  has its own URL.
